# ADR-006: Redis Cache-Aside для Query Handlers

- **Status:** Accepted
- **Date:** 2026-02-23
- **Author:** Backend Team
- **Context:**
  Read-side (`QueryHandler`) в `animals` выполняет повторяющиеся выборки с одинаковыми фильтрами, что увеличивает нагрузку на PostgreSQL при росте трафика.
  В текущей архитектуре (NestJS + CQRS + modular monolith) нужно добавить кэш так, чтобы:
    1. не усложнять write-side,
    2. сохранить независимость QueryHandlers,
    3. иметь прозрачный путь к масштабированию через Redis.

- **Decision:**
  Выбран паттерн **Cache-Aside** с Redis:
    1. QueryHandler сначала пытается прочитать данные из Redis по ключу, построенному как `namespace + sha256(query)`.
    2. При `cache miss` QueryHandler читает данные из БД.
    3. Результат сериализуется и сохраняется в Redis с TTL.
    4. Cache key строится детерминированно из query-параметров (стабильная сериализация + hash), чтобы исключить дублирование ключей при разном порядке полей.

- **Consequences:**
    - **Positive:**
        - Снижение latency и нагрузки на БД для горячих read-запросов.
        - Простой и предсказуемый контроль TTL на уровне запроса.
        - Масштабирование кэша между инстансами API без sticky-сессий.
    - **Negative:**
        - Риск возврата устаревших данных в пределах TTL.
        - Появляется дополнительная инфраструктурная зависимость (Redis).

- **Compliance:**
    - Кэширование применяется только в QueryHandlers.
    - Команды (write-side) не читают кэш и не зависят от Redis.
    - Ключи для фильтруемых списков обязаны формироваться через hash query payload.
