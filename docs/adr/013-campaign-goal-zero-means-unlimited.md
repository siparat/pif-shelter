# ADR-013: `goal = 0` означает неограниченный сбор

- **Status:** Accepted
- **Date:** 2026-03-31
- **Author:** Backend Team

## Context

Для срочных сборов нужна поддержка двух режимов:

1. Сбор с ограниченной целью (`goal > 0`) — закрывается как `SUCCESS` при достижении/превышении цели.
2. Сбор без лимита — когда средства собираются без фиксированной верхней границы.

Ранее в продуктовой формулировке уже использовалось значение `0` как признак неограниченного сбора, но это нужно закрепить архитектурно и единообразно для API, доменной логики и БД.

## Decision

1. Поле `campaigns.goal` хранится в копейках (`integer`).
2. Значение **`goal = 0` интерпретируется как неограниченный сбор**.
3. Автопереход в `SUCCESS` выполняется **только** для сборов с `goal > 0`, когда `collected >= goal`.
4. Для `goal = 0` переход в `SUCCESS` не выполняется автоматически по сумме пополнений.
5. Валидация входных данных для создания/обновления сбора допускает `goal >= 0`.

## Consequences

- **Positive**
    - Простая и детерминированная семантика без дополнительных nullable-полей.
    - Единое правило для API, обработчиков пополнения и UI.
    - Нет миграций по изменению структуры кампаний (используется существующее поле `goal`).

- **Negative**
    - Нельзя отличить «лимит отсутствует» от «лимит не задан по ошибке» без дополнительных продуктовых проверок.
    - Для аналитики и UI нужно явно учитывать отдельный режим `goal = 0`.

## Compliance

- Запрещено трактовать `goal = 0` как «цель достигнута».
- В логике автозакрытия по сумме обязательно проверять `goal > 0` перед сравнением `collected >= goal`.
