# ADR-014: Frontend Development Standards (FSD & PascalCase)

- **Status:** Accepted
- **Date:** 2026-04-14
- **Author:** Frontend Team
- **Context:**
  Для обеспечения масштабируемости и единства стиля в фронтенд-приложениях (админка и др.) требуется зафиксировать архитектурный стандарт.
  Без четких правил фронтенд быстро превращается в "спагетти-код" с перемешанными компонентами и непредсказуемой структурой файлов.
  Нам необходимо решить три ключевые проблемы:
    1.  Размытые границы между бизнес-логикой и UI-компонентами.
    2.  Хаотичное именование файлов и папок (mix из camelCase и kebab-case).
    3.  Отсутствие единого синтаксиса объявлений компонентов (mix из function и arrow functions).

- **Decision:**
  Мы внедряем **Feature-Sliced Design (FSD)** и строгие правила кодинга:
    1.  **Архитектура (FSD):**
        - Разделение кода на слои: `app`, `pages`, `widgets`, `features`, `entities`, `shared`.
        - Соблюдение иерархии импортов: слои выше не могут импортироваться слоями ниже.
        - _Причина:_ Изоляция ответственности и предсказуемость связей.

    2.  **Именование (PascalCase для UI):**
        - Все директории и файлы внутри папок `ui`, содержащие JSX/TSX, пишутся в **PascalCase**.
        - Файлы без JSX (хуки, утилиты, стили) пишутся в **kebab-case**.
        - _Причина:_ Визуальное отличие UI-компонентов от вспомогательной логики.

    3.  **Синтаксис компонентов:**
        - Только функциональные компоненты через **Arrow Functions** (`const Component = () => ...`).
        - _Причина:_ Единство стиля и строгая типизация.

    4.  **Стилизация (Tailwind v4):**
        - Использование Tailwind CSS v4 через Vite-плагин.
        - Объединение классов только через утилиту `cn` (`clsx` + `tailwind-merge`).
        - _Причина:_ Избежание конфликтов специфичности и читаемость динамических классов.

- **Consequences:**
    - **Positive:**
        - Высокая модульность и легкость тестирования отдельных виджетов/фич.
        - Прозрачная навигация по коду благодаря FSD и PascalCase.
    - **Negative:**
        - Оверхед на создание `index.ts` (Public API) для каждого слайса.

- **Compliance:**
    - Код-ревьюверы должны отклонять PR, где UI-компоненты названы в kebab-case.
    - Код-ревьюверы должны отклонять PR, где компоненты объявлены через ключевое слово `function`.
    - Код-ревьюверы должны отклонять PR с прямым использованием `@apply` или конкатенацией строк для классов вместо `cn`.
