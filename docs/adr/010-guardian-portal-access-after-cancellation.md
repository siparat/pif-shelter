# ADR-010: Портальный доступ опекуна после отмены опеки (grace до конца оплаченного периода)

- **Status:** Accepted
- **Date:** 2026-03-21
- **Author:** Backend Team
- **Context:**
  При отмене опеки **без возврата средств** опекун по продукту должен до конца **уже оплаченного** периода подписки сохранять доступ к личному функционалу: ЛК, Telegram-бот («мои животные», карточка, приватные посты), те же проверки, что и у `ACTIVE` опеки.
  Если ориентироваться только на `guardianships.status === ACTIVE`, то сразу после отмены (`CANCELLED`) этот доступ пропадает, хотя деньги за период не возвращались.
  Отдельно: на публичной витрине животного «текущий опекун» должен отображаться только при действующей опеке (`ACTIVE`), иначе бывший опекун в периоде grace блокирует восприятие «животное без опекуна» и мешает оформлению новой опеки (слот в БД при этом уже свободен за счёт partial unique index только на `ACTIVE` / `PENDING_PAYMENT`).

- **Decision:**
    1. **Две колонки в `guardianships`:**
        - `paid_period_end_at` — конец текущего оплаченного биллингового периода; выставляется при первой успешной оплате и **продлевается** при повторном `subscription.succeeded` для уже `ACTIVE` (рекуррентные списания).
        - `guardian_privileges_until` — до какой даты опекун считается имеющим **портальный** доступ после перехода в `CANCELLED`; заполняется при отмене без ожидания возврата (`isRefundExpected === false`), если **до отмены** статус был `ACTIVE`, значением `paid_period_end_at` (если даты нет — колонка остаётся `null`, в лог пишется предупреждение).
    2. **Единое условие «портальный опекун»** (реализация: `guardianshipPortalAccessWhere` в `@pif/database`):
       `status === ACTIVE` **или** (`status === CANCELLED` и `guardian_privileges_until` задано и `> now()`).
    3. **Где применяется предикат:** список «мои опекунства», карточка опекуна по животному, `CanViewPostPolicy`, `ListPostsHandler` для роли опекуна, выбор получателя Telegram-уведомления о новом посте.
    4. **Публичный снимок опеки по животному** (`GetGuardianshipByAnimal`): только `ACTIVE`, без grace — чтобы не показывать бывшего опекуна как «текущего».
    5. **Несколько строк на одно `animalId`:** теоретически возможны `CANCELLED` с ещё действующим `guardian_privileges_until` и новая `ACTIVE` опека. В `PostsProcessor` при выборе опекуна для уведомления о посте используется **приоритет `ACTIVE`** (`ORDER BY`: сначала активная запись), иначе `LIMIT 1` без порядка мог бы выбрать строку бывшего опекуна.
    6. **Централизация правила отмены:** `resolveGuardianPrivilegesUntilForCancel` в модуле guardianship; все пути отмены передают в `repository.cancel` третий аргумент `guardianPrivilegesUntil` явно.

- **Consequences:**
    - **Positive:** Поведение согласовано с ожиданием «оплатил период — пользуешься до конца периода»; новая опека на животное не блокируется; публичное отображение не путает grace с текущей опекой.
    - **Negative:** Два поля вместо одного: `paid_period_end_at` отвечает за биллинг, `guardian_privileges_until` — за снимок прав после отмены; при смене продуктовых правил нужно помнить оба. Любой новый read-путь «как опекун» должен использовать общий предикат, иначе появится расхождение с ботом и постами.

- **Compliance:**
    - Не проверять портальный доступ опекуна только по `status === ACTIVE`.
    - При отмене с возвратом (`isRefundExpected === true`) не выставлять `guardian_privileges_until`.
    - При выборе опекуна по одному `animalId` для side-effect (уведомления) учитывать приоритет `ACTIVE` при нескольких подходящих строках.
