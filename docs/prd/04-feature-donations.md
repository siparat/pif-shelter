# 04. Финансы и Пожертвования (Feature Spec)

Модуль описывает процессы приема денежных средств (эквайринг), управление регулярными подписками и координацию гуманитарной помощи (вещи).

## 1. Типы денежных пожертвований

### 1.1. Разовый донат (One-time Payment)
Пользователь переводит произвольную сумму.
* **Назначение:**
    1.  **Общее:** "На уставную деятельность" (корм, коммуналка).
    2.  **Адресное:** Привязка к `animal_id` (конкретному животному).
    3.  **Сбор (Campaign):** Привязка к `campaign_id` (напр., "Срочная операция для Рекса").
* **Суммы:** Предустановленные кнопки (300, 500, 1000 ₽) + Поле "Своя сумма".
* **Процесс:**
    1.  Выбор суммы и цели.
    2.  Редирект на платежный шлюз (ЮКасса/CloudPayments/Tinkoff — *выбрать провайдера*).
    3.  Возврат на страницу "Спасибо" (Success Page).

### 1.2. Регулярная подписка (Recurrent Payment)
Автоматическое списание средств раз в месяц.
* **Сценарий:** Пользователь ставит галочку "Помогать ежемесячно".
* **Техническая реализация:**
    * Мы НЕ храним данные карт (PCI DSS compliance).
    * Мы сохраняем `payment_token` (токен рекуррента), который возвращает банк после первого успешного платежа.
    * Cron-задача на бэкенде раз в 30 дней инициирует списание по токену.
* **Управление:**
    * Отмена подписки доступна в личном кабинете/интерфейсе платёжного провайдера (PSP) или по ссылке в письме-уведомлении.

### 1.3. Срочные сборы (Campaigns)
Отдельная сущность для форс-мажоров.
* **Атрибуты:** Заголовок, Цель (сумма), Собрано (сумма), Дедлайн, Описание, Фото.
* **Логика:**
    * Прогресс-бар заполняется в реальном времени.
    * **Overfunding (Пересбор):** Если собрали больше (например, 15 000 из 10 000), сбор закрывается, а остаток переносится на баланс "Общие нужды". Об этом должна быть публичная оферта.

## 2. Помощь вещами (In-kind Donations)
Упрощенная модель (без строгого учета склада).

### 2.1. Список нужд (Wishlist)
Админ управляет списком категорий и товаров в админке.
* *Пример:*
    * Категория "Медицина": "Бинты", "Хлоргексидин".
    * Категория "Быт": "Пеленки 60х90", "Средство для мытья пола".
* **Статусы товаров:**
    * 🟢 "Всегда нужно"
    * 🔴 "SOS (Заканчивается)"
    * ⚪ "Временно не нужно" (скрыто).

### 2.2. Заявка на привоз
Пользователь читает на сайте список нужн, и привозит на указанный адрес, его там встретят, без оформления заявки.

## 3. Финансовая прозрачность

### 3.1. Лог транзакций
Публичная таблица на сайте "Отчетность".
* **Поля:** Дата, Сумма, Тип (Разовый/Подписка), Имя (или "Аноним"), Цель.
* **Фильтрация:** По месяцам, по годам.

### 3.2. Документальные отчеты
* **Генерация XLSX:** Админ загружает сканы чеков/актов из ветклиник в админку.
* Система формирует ежемесячный отчет на странице "Документы".

## 4. Статусная модель транзакции (Transaction Lifecycle)

| Статус | Описание |
| :--- | :--- |
| **CREATED** | Запись создана в БД, пользователь ушел на шлюз. |
| **PENDING** | Шлюз обрабатывает платеж. |
| **SUCCESS** | Шлюз прислал webhook: `payment.succeeded`. Деньги зачтены. |
| **FAILED** | Ошибка (недостаточно средств, ошибка банка). |
| **REFUNDED** | Админ сделал возврат средств (редкий кейс). |

## 5. Уведомления (Transaction Emails/Messages)
* **Успешный донат:** "Спасибо, Иван! Ваши 500₽ помогут купить 2 кг корма." (Отправка на Email).
* **Ошибка списания (рекуррент):** "Не удалось списать пожертвование. Проверьте карту." (Отправка в TG бот или Email).
