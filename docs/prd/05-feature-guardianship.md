# 05. Опекунство и Telegram-бот (Feature Spec)

Опекунство — это флагманский продукт приюта. Это подписка на эмоциональную связь с животным и гарантия его сытой жизни.
Основной канал коммуникации с опекуном — **Telegram-бот**. Мы не делаем личный кабинет на сайте, чтобы сэкономить ресурсы разработки и упростить путь пользователя.

## 1. Модель Опекунства

### 1.1. Суть предложения

- **Стоимость:** Фиксированная (3200 ₽/мес) или произвольная (но не менее порога, установленного в админке, например 500 ₽).
- **Что получает опекун:**
    1.  Статус "Опекун" на карточке животного (эксклюзивно). По желанию выводим имя на сайте: "Опекун: Иван".
    2.  Персональные отчеты (фото/видео) раз в месяц прямо в мессенджер.
    3.  Приоритетная запись на прогулку (без общей очереди).

### 1.2. Связь "Животное — Опекун"

- **Правило:** У одного животного может быть только **ОДИН** опекун. (Эксклюзивное право).
    - _UI требование:_ Если у животного уже есть опекун, кнопка на сайте блокируется.
- У одного опекуна может быть **несколько животных** (без ограничений).

## 2. Пользовательский путь (User Flow)

### Шаг 1: Оформление на сайте

Есть два сценария в зависимости от того, зарегистрирован ли пользователь (его email или Telegram уже есть в системе).

**Сценарий А — Новый пользователь (гостевая отправка формы):**

1.  Пользователь выбирает свободное животное и нажимает «Стать опекуном».
2.  Заполняет форму: **имя**, **email**, **Telegram-никнейм** (обязательно, с @).
3.  Фронтенд отправляет **POST /guardianships** (без авторизации) с телом: `animalId`, `name`, `email`, `telegramUsername`.
4.  Бэкенд проверяет: есть ли пользователь с таким email или Telegram.
    - **Если нет** — создаётся аккаунт с ролью «Опекун» (пароль генерируется, данные для входа приходят на почту), создаётся запись опекунства со статусом `PENDING_PAYMENT`, возвращается **201** с `guardianshipId` и `paymentUrl`.
    - **Если да** — возвращается **409** (код `GuardianRequiresAuth`) с сообщением о необходимости войти в аккаунт → переходим к сценарию Б.

**Сценарий Б — Уже зарегистрированный пользователь (после входа):**

1.  После ответа 409 фронт показывает форму входа.
2.  Пользователь авторизуется (email/пароль).
3.  Форма «Стать опекуном» подставляется данными из профиля (имя, email, Telegram — только для отображения, редактируются в личном кабинете).
4.  Фронтенд отправляет **POST /guardianships/authenticated** (с авторизацией) с телом: только `animalId`.
5.  Бэкенд создаёт опекунство со статусом `PENDING_PAYMENT`, возвращает **201** с `guardianshipId` и `paymentUrl`.

**Дальше общий путь:**

6.  Пользователь переходит по `paymentUrl` и оплачивает первый месяц.
7.  **Success Page:** «Спасибо! Теперь [Имя животного] под вашей защитой.»
    - Кнопка **[Перейти в Telegram-бот]** со ссылкой
    - На email опекуна при активации опекунства уходит письмо: **ссылка для отмены** опекунства (одноразовая) и **ссылка на Telegram-бот** — чтобы при перезагрузке страницы человек мог получить ссылку из почты.

### Шаг 2: Активация бота (Onboarding)

**Сценарий А (Основной — Deep Link):**

1.  Пользователь нажимает кнопку на сайте, открывается Telegram.
2.  Нажимает `/start`.
3.  Бот считывает токен из ссылки, находит опекунство/пользователя в БД.
4.  Бот привязывает `chat_id` пользователя к профилю опекуна.
5.  Ответ бота: _"Привет! Я Мила. Спасибо, что ты со мной! Теперь я буду присылать новости сюда."_ (Прикрепляется фото животного).

**Сценарий Б (Резервный — Ручной):**
_Если пользователь закрыл сайт и нашел бота вручную._

1.  Пользователь запускает бота `/start`.
2.  В меню выбирает: "🏁 Я стал опекуном".
3.  Бот предлагает выбрать животное из списка (или ввести имя).
4.  Бот проверяет: совпадает ли `username` текущего пользователя Telegram с тем, который был указан при оплате на сайте.
5.  Если совпадение найдено -> Привязка `chat_id` и успех.
6.  Если нет -> "Не могу найти вашу подписку. Пожалуйста, напишите администратору".

## 3. Функционал Telegram-бота

### 3.1. Команды меню

- `/my_animals` — Список моих подопечных.
    - Ответ: Карусель или список кнопок с именами животных. При нажатии выдает краткую справку (остаток дней до списания, последнее фото).
- `/report` — Запросить статус (Автоответ: "Волонтеры сейчас заняты уходом за животными. Ваш запрос передан, ответим при первой возможности").
- `/help` — Справка: контакты приюта, геолокация, правила посещения. Контент для `/help` собирается до разработки бота и хранится в БД; при сидинге значения берутся из .env.
- `/unsubscribe` — Управление подпиской (отмена).

### 3.2. Рассылка отчетов (Push Flow)

Киллер-фича для удержания (Retention).

1.  **Волонтер (в Админке):**
    - Открывает карточку животного.
    - Нажимает "Отправить отчет опекуну".
    - Загружает медиа (фото/видео) и текст.
2.  **Бэкенд:**
    - Находит `chat_id` единственного опекуна.
    - Отправляет сообщение через Bot API (метод `sendMediaGroup` для красивой галереи).

### 3.3. Уведомления о платежах

- **Успех:** "Ваш взнос за [Имя животного] получен. Спасибо!"
- **Ошибка:** "Не удалось продлить опеку над [Имя животного]. Пожалуйста, обновите карту по ссылке [Link]. Подписка будет ждать еще 3 дня."

## 4. Техническая реализация

### 4.1. Безопасность

- Ссылка для привязки бота **одноразовая**: после первой успешной активации она «сгорает», чтобы нельзя было передать доступ другому человеку.
- Если пользователь блокирует бота (`forbidden`), система помечает это в БД флагом (например `telegram_unreachable`), чтобы не слать сообщения в недоступный чат.

### 4.2. Письмо через 24 часа, если бот не активирован

- Если в течение **24 часов** после активации опекунства опекун не привязал Telegram-бота, на email уходит напоминание: без привязки бота мы не можем отправлять отчёты по [Имя животного]; в письме снова даётся ссылка на бота (или инструкция получить новую в ЛК).

### 4.3. Флоу оформления опекунства и оплаты (API)

**1. Старт без авторизации (новый или «занятый» пользователь)**

- **Запрос:** `POST /guardianships`
  Тело: `{ animalId, name, email, telegramUsername }` (все поля обязательны).
- **Логика бэкенда:**
    - Поиск пользователя по `email` и по `telegramUsername`.
    - Если пользователь **найден** → ответ **409** (код `GuardianRequiresAuth`): «Войдите в аккаунт»; фронт показывает форму входа, после входа использует шаг 2.
    - Если **не найден** → создаётся пользователь (роль GUARDIAN, позиция «Опекун», пароль генерируется; на email уходит письмо с данными для входа), создаётся запись в `guardianships` со статусом `PENDING_PAYMENT` и уникальным `subscriptionId`, возвращается **201** с `guardianshipId` и `paymentUrl`.

**2. Старт с авторизацией (уже залогинен)**

- **Запрос:** `POST /guardianships/authenticated` (нужен заголовок/куки авторизации).
  Тело: `{ animalId }`.
- **Логика:** создаётся запись опекунства в `PENDING_PAYMENT` с `subscriptionId`, возвращается **201** с `guardianshipId` и `paymentUrl`. Имя, email и Telegram берутся из профиля пользователя.

**3. Оплата и вебхук**

- Пользователь переходит по `paymentUrl` и оплачивает первый период у платёжного провайдера.
- Провайдер при успешной оплате вызывает вебхук бэкенда (в тестах — моковый endpoint).
- **Мок вебхука:** `POST /guardianships/webhooks/payment`
  Тело: `{ subscriptionId, event: "payment.succeeded" }`.
  Бэкенд находит опекунство по `subscriptionId`, переводит статус в `ACTIVE`, публикует доменное событие `GuardianshipActivatedEvent`. По этому событию опекуну отправляется письмо: ссылка для отмены опекунства (одноразовый токен) и ссылка на Telegram-бота. Через 24 часа при неактивированном боте — напоминание (см. п. 4.2).

**4. Проверка статуса после оплаты**

- Фронт может опросить `GET /guardianships/by-animal/:animalId`, чтобы убедиться, что статус стал активным, и получить данные для Success Page (имя опекуна, животное и т.д.).

**5. Отмена опекунства**

- **По ссылке из письма (без авторизации):** пользователь переходит по ссылке (например `{frontend}/cancel-guardianship?token=...`), фронт отправляет **POST /guardianships/cancel-by-token** с телом `{ token }`. Опекунство переводится в `CANCELLED`, токен в письме одноразовый (после отмены обнуляется).
- **Админ:** **POST /guardianships/cancel** с телом `{ guardianshipId }` (требуется роль ADMIN).

## 5. Обработка отмены (Churn)

Процесс максимально упрощен, чтобы не вызывать негатив.

1.  Пользователь выбирает `/unsubscribe` -> [Имя животного].
2.  Подтверждает действие.
3.  **Бэкенд:** Немедленно удаляет токен рекуррентного списания (подписка дорабатывает оплаченный период).
4.  **Бот:** "Подписка отменена. Жаль расставаться. [Имя животного] будет скучать."
